BLE 開發指南

本指南介紹如何使用 IPRO SDK 的 BLE 協定棧開發藍牙低功耗應用。SDK 採用 Zephyr BLE Host Stack 搭配 RivieraWaves 預建構控制器，提供完整的 BLE 5.4 功能支援。

協定棧架構

BLE 協定棧採用 Host + Controller 雙層架構，透過片上 HCI 介面通訊（無 UART HCI 傳輸）：

Application Layer
GAP / GATT Services / Profiles
BLE Host (Zephyr-based)
GAP / GATT / L2CAP / SMP / ISO
HCI (On-Chip)
ipro_hci_wrapper
BLE Controller (RivieraWaves)
Link Layer / PHY (1M/2M/Coded)

核心概念

  • Zephyr BLE Host：即使在 FreeRTOS 上也使用 Zephyr BLE Host Stack，應用程式透過標準 bt_* API 操作
  • RivieraWaves 控制器：以預建構靜態庫形式提供，負責 Link Layer 與 PHY 管理
  • 片上 HCI：Host 與 Controller 之間使用片上 HCI 通訊，無需 UART HCI 傳輸層
  • 回調驅動：所有非同步事件（連線、斷線、通知）透過回調函式處理

控制器變體

控制器採用統一連線池架構，linkN 中的 N 表示同時連線數。每個連線槽可作為 Central 或 Peripheral 使用（任意角色組合）。

Kconfig 配置

在應用程式的 .config 中透過 CONFIG_BTBLECONTROLLER_LIB 選擇變體：

.config 變體選擇 
# 直接指定變體名稱
CONFIG_BTBLECONTROLLER_LIB="link2"

# 或使用布林旗標組合（自動拼接後綴）
CONFIG_BTBLECONTROLLER_LIB="link2"
CONFIG_BTBLECONTROLLER_LE_AUDIO=y     # 加上 _leaudio
CONFIG_BTBLECONTROLLER_BT_54=y        # 加上 _53_54（5.4 隱含 5.3）
# 結果變體：link2_leaudio_53_54

預建構庫位於：components/wireless/bluetooth/btblecontroller_ipro7_<variant>/

API 模式

應用程式使用 Zephyr 風格的回調驅動 BLE API。舊版 RivieraWaves ke_msg 訊息傳遞機制僅限控制器內部使用，不對應用程式暴露。

GAP：廣播與掃描

GAP 廣播操作 
#include <bluetooth/bluetooth.h>
#include <bluetooth/conn.h>

/* 廣播資料 */
static const struct bt_data ad[] = {
    BT_DATA_BYTES(BT_DATA_FLAGS,
                  BT_LE_AD_GENERAL | BT_LE_AD_NO_BREDR),
    BT_DATA_BYTES(BT_DATA_UUID16_ALL,
                  0x0d, 0x18),  /* Heart Rate Service */
};

static const struct bt_data sd[] = {
    BT_DATA(BT_DATA_NAME_COMPLETE, "IPRO Device", 11),
};

/* 開始可連線廣播 */
bt_le_adv_start(BT_LE_ADV_CONN_NAME,
                ad, ARRAY_SIZE(ad), sd, ARRAY_SIZE(sd));

/* 停止廣播 */
bt_le_adv_stop();

/* 開始掃描（Central 角色） */
bt_le_scan_start(BT_LE_SCAN_ACTIVE, scan_cb);

連線回調

連線管理 
static void connected(struct bt_conn *conn, u8_t err)
{
    if (err) {
        printk("Connection failed (err %u)\n", err);
        return;
    }
    printk("Connected\n");
}

static void disconnected(struct bt_conn *conn, u8_t reason)
{
    printk("Disconnected (reason %u)\n", reason);
}

static struct bt_conn_cb conn_callbacks = {
    .connected = connected,
    .disconnected = disconnected,
};

void app_init(void)
{
    bt_conn_cb_register(&conn_callbacks);
    bt_enable(bt_ready);
}

GATT：服務註冊

靜態 GATT 服務定義 
#include <bluetooth/gatt.h>
#include <bluetooth/uuid.h>

/* 自訂 UUID */
static struct bt_uuid_128 my_svc_uuid = BT_UUID_INIT_128(
    0xf0, 0xde, 0xbc, 0x9a, 0x78, 0x56, 0x34, 0x12,
    0x78, 0x56, 0x34, 0x12, 0x78, 0x56, 0x34, 0x12);

static struct bt_uuid_128 my_char_uuid = BT_UUID_INIT_128(
    0xf1, 0xde, 0xbc, 0x9a, 0x78, 0x56, 0x34, 0x12,
    0x78, 0x56, 0x34, 0x12, 0x78, 0x56, 0x34, 0x12);

static uint8_t my_value[256];

/* 讀取回調 */
static ssize_t read_my_char(struct bt_conn *conn,
                            const struct bt_gatt_attr *attr,
                            void *buf, u16_t len, u16_t offset)
{
    return bt_gatt_attr_read(conn, attr, buf, len, offset,
                             my_value, sizeof(my_value));
}

/* 使用巨集定義服務（編譯期靜態註冊） */
BT_GATT_SERVICE_DEFINE(my_svc,
    BT_GATT_PRIMARY_SERVICE(&my_svc_uuid),
    BT_GATT_CHARACTERISTIC(&my_char_uuid.uuid,
                           BT_GATT_CHRC_READ | BT_GATT_CHRC_NOTIFY,
                           BT_GATT_PERM_READ,
                           read_my_char, NULL, my_value),
    BT_GATT_CCC(NULL, BT_GATT_PERM_READ | BT_GATT_PERM_WRITE),
);

/* 發送 GATT 通知 */
bt_gatt_notify(conn, &my_svc.attrs[1], my_value, sizeof(my_value));
提示：舊版 ipro_ble_api.h 與 RivieraWaves ke_msg API 已棄用。所有新應用程式應使用 Zephyr bt_* API。

LE Audio 生態系統

SDK 支援完整的 LE Audio 功能，需使用 _leaudio 後綴的控制器變體。標頭檔位於 bluetooth/audio/。

LE Audio Profiles

Profile 全名 功能
BAP Basic Audio Profile 音訊串流管理（單播/廣播）
CAP Common Audio Profile 協調設備集合的音訊操作
VCP Volume Control Profile 音量控制
MCP Media Control Profile 媒體播放控制
CSIP Coordinated Set Identification Profile 協調設備識別（如左右耳機配對）

LC3 編解碼管線

LE Audio 使用 LC3 (Low Complexity Communication Codec) 作為強制編解碼器，支援 32-320 kbps 位元率，音訊延遲低於 20ms。音訊資料透過 Isochronous Channels (CIS/BIS) 傳輸。

配置範例

.config LE Audio 配置 
# 啟用 LE Audio 控制器
CONFIG_BTBLECONTROLLER_LIB="link2"
CONFIG_BTBLECONTROLLER_LE_AUDIO=y
CONFIG_BTBLECONTROLLER_BT_54=y
# 結果變體：link2_leaudio_53_54

# 啟用 LE Audio Host 功能
CONFIG_BT_AUDIO=y
CONFIG_BT_BAP_UNICAST_SERVER=y
CONFIG_BT_ASCS=y
CONFIG_BT_PACS=y
CONFIG_BT_VCP_VOL_REND=y
CONFIG_BT_CSIP_SET_MEMBER=y

PAwR 與 ESL

BLE 5.4 引入 PAwR（Periodic Advertising with Responses），讓 broadcaster 能在固定 subevent 時槽中收集 observer 的回應，無需建立一對一連線。SDK 在 host 與 controller 兩端皆已支援 PAwR，是 IPRO 大規模感測器與 ESL 應用的核心技術。

典型架構

  • Gateway / Broadcaster：每個 PA interval 廣播一次 PAwR train，內含多個 subevent；subevent 0 通常作為 provisioning channel，subevent 1+ 分配給已配對的 sensor
  • Sensor / Observer：依分配到的 subevent 在精確時槽喚醒、解析 broadcaster 命令、回傳感測資料，其餘時間深度睡眠
  • 無連線：sensor 數量可遠超過控制器連線上限（典型 32 sensors / interval）

關鍵 API

PAwR Host API 
/* Broadcaster：建立 PAwR train */
struct bt_le_per_adv_param param = {
    .interval_min   = BT_GAP_PER_ADV_SLOW_INT_MIN,  /* 1s */
    .interval_max   = BT_GAP_PER_ADV_SLOW_INT_MAX,
    .num_subevents  = 32,
    .subevent_interval = 0x10,        /* 20 ms */
    .response_slot_delay  = 0x05,
    .response_slot_spacing = 0x05,
    .num_response_slots  = 1,
};
bt_le_per_adv_set_param(adv, &param);
bt_le_per_adv_set_subevent_data(adv, num_subevents, subevent_data);
bt_le_per_adv_start(adv);

/* Observer：同步 PA train，回傳 response */
bt_le_per_adv_sync_create(...);
bt_le_per_adv_set_response_data(sync, &response_params, &response_buf);
已知問題：bt_le_per_adv_set_response_data() 在 subevent 1+ 上目前回傳 -5（EIO）— 詳見 PHM Gateway session 紀錄。Provisioning channel（subevent 0）已驗證可正常運作。

ESL（Electronic Shelf Label）

SDK 同時提供傳統 ESL（基於連線）與 PAwR 版本：

低功耗 BLE

IPRO7 支援多級低功耗模式，BLE 應用可在 advertising / connection 間隙進入 PDS（Power Down Sleep）以達成 µA 等級的平均電流。

PDS Level 31 — 深度睡眠

  • OCRAM 全保留：IPRO7 OCRAM 256 KB 全部 bank 預設保留，無需 PDS_RAM_Config()
  • BLE Exchange Memory：位於 OCRAM 頂端（0x1104_0000~），不佔用 retention 區
  • BZ Controller：pdsCtl4.BzRst=1, BzGateClk=1 讓 PDS 在睡眠時 reset + gate BLE controller 時脈，喚醒後重新初始化
  • RF 自動關斷：pdsCtlRfSel=1（在 hal_pm.c）讓 PDS 硬體在睡眠時關閉 RF analog blocks，從 80 µA 降至 ~10 µA
  • turnoffDLL=0：cpuRst=1 模式下 post-WFI 不執行 DLL 重啟流程

Cold Boot with Detection

實測 cpuRst=0（純 WFI 恢復）在 BLE controller 模式下不可用；正確方案是 cpuRst=1 + RSV0=0 + RSV2=flag 在 cold boot 中偵測喚醒原因，於 bootrom 之後跳回 BLE state。延遲約 100 ms — 適合稀疏喚醒週期，BLE 連線維持仍需配合 sleep clock 校正。

低功耗 BLE 應用

  • apps/bluetooth/ble/ipro_ble_lowpower_test/ — PDS31 全套驗證 app，包含 RF reinit、register save/restore、margin 量測。Phase 0–4 已實作，PDS31 已硬體驗證
  • apps/bluetooth/esl/ipro_esl_sensor/ — PAwR 模式下的 sensor，每個 subevent 喚醒、回應後立刻深睡
  • apps/turnkey/ipro7_pircam/ — PIR 觸發喚醒，AI 模型常駐 PSRAM 跨睡眠週期
提示：修改 LP/PDS 配置前先排除應用層程式錯誤；多數情況下「電流變高」是 task 沒進 idle 或 timer 沒釋放，而非 LP 設定本身。

應用框架

BLE 應用採用 Mediator 設計模式，將複雜的藍牙功能分解為可重用的元件：

元件 角色 說明
Mediator 協調者 管理 Applet 之間的互動與生命週期，負責整體流程控制
Applets 可重用元件 封裝特定功能（如廣播管理、GATT 服務、音訊串流），可在不同應用間共用
Intermediaries 協議橋接 在不同協議層之間轉換資料格式，例如將應用層命令轉為 BLE GAP/GATT 操作

此框架主要用於 LE Audio 應用（如 GAF、BAP、VCP、MCP、CAP），將複雜的音訊管理邏輯模組化。

預建構函式庫

components/wireless/bluetooth/build_all_prebuilts.sh all

# 輸出位置
