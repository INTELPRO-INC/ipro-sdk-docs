Matter & Thread Development Guide

IPRO SDK supports Matter 1.4, the unified smart home protocol, and Thread 1.2 low-power mesh networking, enabling your devices to seamlessly interoperate with Apple Home, Google Home, Amazon Alexa, and other ecosystems.

Overview

Matter is the unified smart home standard driven by the Connectivity Standards Alliance (CSA), designed to solve interoperability issues between IoT devices. IPRO SDK provides comprehensive Matter and Thread support:

Matter 1.4

  • CSA unified smart home standard
  • Multi-ecosystem interoperability
  • Secure commissioning and control
  • Data Model / Interaction Model

Thread 1.2

  • IEEE 802.15.4 low-power mesh network
  • Self-healing topology
  • Native IPv6 support
  • FTD / MTD / RCP device types

Multi-Transport

  • Matter over Thread (low power)
  • Matter over WiFi (high bandwidth)
  • BLE commissioning channel
  • Border Router bridging

Matter Architecture

The Matter protocol is built on top of IP networking, using Thread or WiFi as the transport layer, and BLE for device commissioning:

Matter Data Model

Matter uses a hierarchical data model to describe device capabilities:

Concept Description Example
Node An addressable Matter device A smart light bulb
Endpoint A functional unit on a Node Endpoint 0 (Root), Endpoint 1 (Light)
Cluster A collection of related features OnOff, LevelControl, ColorControl
Attribute A readable/writable state value OnOff.onOff, LevelControl.currentLevel
Command An executable action OnOff.Toggle(), LevelControl.MoveToLevel()

Interaction Model

Matter devices communicate through the following interaction patterns:

  • Read -- Read attribute values (e.g., query if a light is on)
  • Write -- Write attribute values (e.g., set brightness level)
  • Invoke -- Invoke commands (e.g., Toggle switch)
  • Subscribe -- Subscribe to attribute change notifications (e.g., monitor temperature changes)

Commissioning Flow

Matter uses BLE as the commissioning channel. The process is as follows:

  1. Device boots into commissioning mode and starts BLE advertising
  2. Phone app scans the QR Code or enters the pairing code
  3. Secure connection established via PASE (Passcode-Authenticated Session Establishment)
  4. App sends Thread/WiFi network credentials to the device
  5. Device joins the IP network and establishes CASE (Certificate-Authenticated Session Establishment)
  6. Device is added to the Matter Fabric and can be controlled

Matter over Thread

Thread is a mesh networking protocol designed for low-power IoT devices. Matter over Thread is ideal for battery-powered sensors, switches, and lighting devices.

Thread uses IEEE 802.15.4 radio, which operates in the 2.4 GHz ISM band with a different modulation scheme than WiFi/BLE. IPRO7 supports 802.15.4, WiFi, and BLE radios simultaneously.

Thread Network Topology

Device Roles

Role Type Description
Leader FTD Manages routing table and address allocation; one per partition
Router FTD Forwards packets, maintains routing tables, supports child devices
End Device MTD Communicates only with parent router; does not forward packets
Sleepy End Device MTD Sleeps most of the time; ideal for battery-powered devices
Border Router FTD Bridges Thread and WiFi/Ethernet networks

Typical Use Cases

  • Smart lights and luminaires (Router / End Device)
  • Door/window sensors (Sleepy End Device)
  • Temperature and humidity sensors (Sleepy End Device)
  • Smart switches and plugs (Router)
  • Motion sensors (Sleepy End Device)

Matter over WiFi

Matter over WiFi provides direct IP connectivity through WiFi, suitable for devices requiring higher bandwidth. No Border Router is needed to connect to the home network.

Use Cases

  • Smart cameras (require high-bandwidth video streaming)
  • Smart displays
  • Smart speakers
  • Home appliances (refrigerators, washing machines, etc.)
  • Devices requiring fast response times

Matter over Thread vs WiFi Comparison

Feature Thread WiFi
Power Consumption Very low (battery-friendly) Higher (requires stable power)
Bandwidth 250 kbps Hundreds of Mbps
Network Topology Mesh (self-healing) Star (AP-dependent)
Border Router Required Not required
Max Nodes 250+ (mesh scaling) Limited by AP capacity
Ideal Devices Sensors, switches, lights Cameras, displays, speakers

OpenThread

IPRO SDK integrates Google's open-source OpenThread implementation, providing a full Thread protocol stack. Source code is located at components/wireless/thread/.

OpenThread Component Structure 
components/wireless/thread/
├── CMakeLists.txt              # Build configuration
├── Kconfig                     # Device type selection (FTD/MTD/RCP)
├── openthread/                 # OpenThread open-source stack
│   ├── src/                    # Core implementation
│   │   ├── core/               #   Thread protocol core
│   │   └── cli/                #   CLI command interface
│   ├── include/                # Public API headers
│   └── examples/               # Examples and platform ports
└── openthread_port/            # IPRO7 platform abstraction layer
    └── Kconfig                 #   Platform-specific config options

Device Types

Select the OpenThread device type via Kconfig:

Device Type Configuration 
# Full Thread Device -- can route, can become Leader/Router
CONFIG_OPENTHREAD_FTD=y

# Minimal Thread Device -- End Device only, saves power and memory
CONFIG_OPENTHREAD_MTD=y

# Radio Co-processor -- 802.15.4 radio controlled by external host
CONFIG_OPENTHREAD_RADIO=y

OpenThread CLI

With CONFIG_OPENTHREAD_CLI=y enabled, you can use ot commands via serial port for development and debugging:

Common OpenThread CLI Commands 
# Create a new Thread network
ot dataset init new
ot dataset networkname MyThreadNet
ot dataset channel 15
ot dataset panid 0x1234
ot dataset commit active
ot ifconfig up
ot thread start

# Query status
ot state                  # Show role: disabled/detached/child/router/leader
ot ipaddr                 # Show IPv6 addresses
ot neighbor table         # Show neighbor table
ot router table           # Show router table (FTD only)

# Network diagnostics
ot ping <ipv6_address>    # Ping another device
ot child table            # Child device list
ot parent                 # Parent device info

Commissioner and Joiner

Thread uses the Commissioner/Joiner mechanism to securely add new devices to the network:

Commissioner Operations (device already in network) 
# Start Commissioner
ot commissioner start

# Allow new devices to join (* accepts any device, J01NME is the pre-shared key)
ot commissioner joiner add * J01NME

# Check Commissioner state
ot commissioner state
Joiner Operations (new device joining) 
# Join using pre-shared key
ot joiner start J01NME

# After join success message
ot ifconfig up
ot thread start

Configuration

Matter Configuration

Set Matter options in the application's defconfig or .config file:

Matter Configuration Options 
# Enable Matter
CONFIG_MATTER_ENABLE=y
CONFIG_COMPONENTS_MATTER_ENABLE=y

# Device information
CONFIG_MATTER_VENDOR_ID=0xFFF1          # CSA test Vendor ID
CONFIG_MATTER_PRODUCT_ID=0x8000         # Product ID
CONFIG_MATTER_DEVICE_TYPE=256           # Device type (256 = On/Off Light)

# Required dependencies
CONFIG_FREERTOS_SUPPORT=y               # FreeRTOS
CONFIG_COMPONENTS_MBEDTLS_ENABLE=y      # Security layer
CONFIG_COMPONENTS_LWIP_ENABLE=y         # IP network stack

Thread / OpenThread Configuration

OpenThread Configuration Options 
# Enable Thread
CONFIG_THREAD_ENABLE=y
CONFIG_OPENTHREAD_ENABLE=y
CONFIG_IEEE802154_ENABLE=y

# Device type (choose one)
CONFIG_OPENTHREAD_FTD=y                 # Full Thread Device
# CONFIG_OPENTHREAD_MTD=y              # Minimal Thread Device

# CLI interface
CONFIG_OPENTHREAD_CLI=y

# 802.15.4 radio
CONFIG_IEEE802154_DEFAULT_TXPOWER=20    # TX power (dBm)

# Feature enables (select as needed)
CONFIG_OPENTHREAD_COMMISSIONER=y        # Commissioner role
CONFIG_OPENTHREAD_JOINER=y              # Joiner role
CONFIG_OPENTHREAD_DHCP6_CLIENT=y        # DHCPv6 Client
CONFIG_OPENTHREAD_DHCP6_SERVER=y        # DHCPv6 Server
CONFIG_OPENTHREAD_DNS_CLIENT=y          # DNS Client
CONFIG_OPENTHREAD_BORDER_ROUTER=y       # Border Router functionality

# Memory configuration
CONFIG_OPENTHREAD_NUM_MESSAGE_BUFFERS=44
CONFIG_OPENTHREAD_MESSAGE_BUFFER_SIZE=128
CONFIG_OPENTHREAD_MLE_MAX_CHILDREN=16

# Log level (0=NONE, 1=CRIT, 2=WARN, 3=NOTE, 4=INFO, 5=DEBG)
CONFIG_OPENTHREAD_LOG_LEVEL=3
Disabling unused features can significantly save memory: Commissioner/Joiner requires ~150KB (DTLS/SSL), Border Router ~30KB, ECDSA ~15KB, SRP Client ~20KB.

Selecting Matter over Thread vs WiFi

Transport Layer Selection 
# Matter over Thread (low-power devices)
CONFIG_THREAD_ENABLE=y
CONFIG_OPENTHREAD_ENABLE=y
CONFIG_IEEE802154_ENABLE=y
# CONFIG_WIFI6 is not set

# Matter over WiFi (high-bandwidth devices)
CONFIG_WIFI6=y
CONFIG_COMPONENTS_LWIP_ENABLE=y
# CONFIG_THREAD_ENABLE is not set

Development Workflow

1. Choose a Reference Application

Select an appropriate reference application as your starting point based on your target device type:

  • Lighting devices -- Start with apps/matter/lighting-app
  • Multi-function devices -- Start with apps/matter/all-clusters-app
  • Thread network development -- Start with apps/ipro_thread_ftd_cli

2. Build Firmware

Build Steps 
# Build Matter Lighting App
make -C apps/matter/lighting-app

# Build Thread FTD CLI
make -C apps/ipro_thread_ftd_cli

# Clean build
make -C apps/ipro_thread_ftd_cli clean

3. Flash and Test

Flash Firmware 
# Flash to development board
tools/ipro_iot_tool_lite/ipro_iot_tool_lite \
    --chipname ipro7 \
    --port /dev/cu.usbserial-2120 \
    --firmware apps/ipro_thread_ftd_cli/build/build_out/ipro_thread_ftd_cli_IPRO7.bin \
    --baudrate 921600 \
    --flash-pin 0x02

# Connect serial monitor
python3 tools/serial_monitor.py /dev/cu.usbserial-2120

4. Matter Commissioning Test

  1. Use Google Home App, Apple Home App, or chip-tool for commissioning
  2. Scan the device QR Code or manually enter the pairing code
  3. Once the device joins the Matter Fabric, it can be controlled
Testing with chip-tool (Development) 
# Commissioning
chip-tool pairing ble-thread <node-id> <thread-dataset> <setup-pin-code> <discriminator>

# Control On/Off
chip-tool onoff toggle <node-id> 1

# Read attributes
chip-tool onoff read on-off <node-id> 1

# Set brightness
chip-tool levelcontrol move-to-level <level> 0 0 0 <node-id> 1
