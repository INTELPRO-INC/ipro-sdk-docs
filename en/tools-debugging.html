Tools & Debugging

The IPRO SDK provides an integrated toolchain covering the complete development cycle: build, flash, monitor, and debug.

Overview

A typical embedded development workflow follows this cycle:

Development Cycle 
Edit Code → Build → Flash → Monitor → Debug → Repeat

The SDK provides the following tools in the tools/ directory:

Tool Path Purpose
Flash Tool tools/ipro_iot_tool_lite/ Flash firmware to the development board
Serial Monitor tools/serial_monitor.py Serial communication and log capture
Board Test tools/board_test.sh Automated build, flash, and test
Crash Parser tools/parse_backtrace.py Convert crash addresses to source locations
Size Analysis tools/size_analysis.sh Analyze firmware memory usage
OpenOCD Config tools/openocd/ JTAG debug interface configuration

Flash Tool

The IPRO SDK uses ipro_iot_tool_lite to flash compiled firmware to the development board's Flash memory.

Basic Usage

Flash Firmware 
tools/ipro_iot_tool_lite/ipro_iot_tool_lite \
    --chipname ipro7 \
    --port /dev/cu.usbserial-2120 \
    --firmware apps/bluetooth/ipro_ble_remote/build/build_out/ipro_ble_remote_IPRO7.bin \
    --baudrate 921600 \
    --flash-pin 0x02

Parameters

Parameter Description Example
--chipname Target chip model ipro7
--port Serial device path /dev/cu.usbserial-2120 (macOS)
/dev/ttyUSB0 (Linux)
--firmware Firmware binary path apps/.../build/build_out/*.bin
--baudrate Flash transfer baud rate 921600
--flash-pin Flash control pin 0x02

Entering Download Mode

The board must be in download mode before flashing. Send the reboot_dl command via serial:

Enter Download Mode 
# Send reboot_dl command via serial monitor
python3 tools/serial_monitor.py /dev/cu.usbserial-2120 --send reboot_dl --timeout 1
⚠️ On macOS, you must use /dev/cu.* device paths. Do not use /dev/tty.* as it will block indefinitely waiting for the DCD signal.

Serial Monitor

serial_monitor.py is a Python3 termios-based serial monitor that requires no external dependencies such as pyserial.

Basic Monitoring

Basic Serial Monitoring 
# Default baud rate: 115200
python3 tools/serial_monitor.py /dev/cu.usbserial-2120

Send Commands and Wait for Output

Send Command and Wait for Pattern 
# Send reboot and wait for test results
python3 tools/serial_monitor.py /dev/cu.usbserial-2120 \
    --send reboot \
    --timeout 30 \
    --wait-for "ALL TESTS PASSED|SOME TESTS FAILED"

Parameters

Parameter Description
PORT Serial device path (positional argument)
--send CMD Command to send after connecting
--timeout SEC Timeout in seconds
--wait-for PATTERN Regex pattern to wait for (supports | for multiple patterns)
--log FILE Save output to a log file

Board Shell Commands

Command Description
reboot Normal board reset
reboot_dl Reset and enter download mode (for flashing)

Automated Testing

board_test.sh integrates the full build, flash, and test workflow, suitable for both CI/CD and local development.

Full Test Cycle

Full Build, Flash, and Test 
# Full cycle: build → flash → serial capture → result parsing
tools/board_test.sh --project apps/ipro_unit_test

Options

Option Description Default
--project PATH Project directory path apps/ipro_unit_test
--port PORT Serial device path /dev/cu.usbserial-2120
--skip-build Skip the build step -
--skip-flash Skip the flash step -
--timeout SEC Serial monitor timeout in seconds 30
--report DIR Generate test report in the specified directory Disabled

Exit Codes

Exit Code Meaning
0 All tests passed (ALL TESTS PASSED)
1 Some tests failed (SOME TESTS FAILED)
2 Timeout - no test result detected

Quick Retest (No Rebuild or Reflash)

Quick Retest 
# Reboot the board and capture test results only
python3 tools/serial_monitor.py /dev/cu.usbserial-2120 \
    --send reboot \
    --timeout 30 \
    --wait-for "ALL TESTS PASSED|SOME TESTS FAILED"
💡 Set CONFIG_TEST_AUTO_RUN=y in your .config file to have tests run automatically on boot without manual triggering.

GDB Debugging

The IPRO SDK supports JTAG debugging through OpenOCD + GDB.

Start OpenOCD

Start OpenOCD Server 
# Use the SDK-provided OpenOCD configuration
openocd -f tools/openocd/openocd_ipro6.cfg

Connect GDB

Launch GDB and Connect 
# Use the SDK toolchain's GDB
/opt/toolchain/riscv_ipro7/bin/riscv64-unknown-elf-gdb \
    apps/ipro7_demo/build/build_out/ipro7_demo_IPRO7.elf

# Inside GDB, connect to OpenOCD
(gdb) target remote :3333
(gdb) monitor reset halt
(gdb) load

Common GDB Commands

Command Description
b main Set breakpoint at main function
b file.c:42 Set breakpoint at a specific file and line
c Continue execution
s / n Step into / Step over
p variable Print variable value
x/16xw 0x11010000 Examine memory (16 words in hex)
bt Show backtrace / call stack
info reg Display register state

Crash Analysis

When the firmware encounters an exception, the exception handler outputs register state and stack backtrace information. parse_backtrace.py converts these addresses into source file names and line numbers.

Usage

Parse from File 
# Parse crash log from a file
python3 tools/parse_backtrace.py \
    /opt/toolchain/riscv_ipro7/bin/riscv64-unknown-elf- \
    apps/ipro7_demo/build/build_out/ipro7_demo_IPRO7.elf \
    backtrace.txt
Parse from Pipe / Stdin 
# Pipe from clipboard or file
cat backtrace.txt | python3 tools/parse_backtrace.py \
    /opt/toolchain/riscv_ipro7/bin/riscv64-unknown-elf- \
    apps/ipro7_demo/build/build_out/ipro7_demo_IPRO7.elf

Parameters

Parameter Description
toolchain_prefix Toolchain prefix path (including trailing -)
elf_file Corresponding ELF file (with debug symbols)
backtrace_file Crash log file (optional, reads from stdin if omitted)
💡 Parsing requires the ELF file (.elf) with debug symbols, not the .bin file used for flashing. ELF files are located in the build/build_out/ directory.

Build Size Analysis

size_analysis.sh analyzes firmware memory usage, helping you understand how much Flash and RAM each module consumes.

Usage

Memory Size Analysis 
# Analyze memory usage for a project
./tools/size_analysis.sh ipro7_demo

The analysis report includes:

  • text - Code section (stored in Flash)
  • data - Initialized data (occupies both Flash and RAM)
  • bss - Uninitialized data (occupies RAM only)
  • Memory usage breakdown per component and library

Results are written to SIZE_ANALYSIS.md in the project directory.

IPRO7 Memory Layout

Region Size Base Address
Flash 1 MB 0x13000000
PSRAM 8 MB 0x1e000000
OCRAM 256 KB 0x11010000
HBNRAM 4 KB 0x20090000

CI/CD Pipeline

The IPRO SDK uses GitLab CI for continuous integration and deployment, configured in .gitlab-ci.yml.

Pipeline Stages

CI/CD Stages 
prepare → build → test → security → notify → merge
Stage Description
prepare Verify build environment (CMake, Ninja, Python3, Git)
build Compile all target applications (BLE, general, etc.)
test Run automated tests
security Security scanning
notify Send build result notifications (Teams webhook)
merge Automated merge handling

CI Environment

  • Toolchain path: /opt/toolchain/riscv_ipro7/
  • CMake: Bundled Linux x86-64 binary at tools/cmake/
  • Parallel builds: 4 parallel jobs by default (CMAKE_BUILD_PARALLEL_LEVEL=4)
  • Cache strategy: Keyed by branch and job name
⚠️ The bundled CMake at tools/cmake/ is a Linux x86-64 ELF binary and cannot run on macOS. Install CMake via brew install cmake for local macOS development.

macOS Development Notes

There are several platform-specific considerations when developing on macOS.

Serial Ports

⚠️ On macOS, always use /dev/cu.* instead of /dev/tty.*. The tty.* devices wait for the DCD (Data Carrier Detect) signal, causing programs to block indefinitely.
Operating System Serial Device Path
macOS /dev/cu.usbserial-2120
Linux /dev/ttyUSB0

Common Issues and Solutions

Issue Cause Solution
Serial program hangs Using /dev/tty.* Switch to /dev/cu.*
Incomplete cat output macOS buffering issues Use serial_monitor.py
"Resource busy" error stty -f holding the port Close the holding process; use Python approach
CI CMake won't run Bundled binary is Linux ELF Use brew install cmake

Recommended Serial Port Flags

Python Serial Port Example 
import os

# Correct way to open serial ports on macOS
fd = os.open("/dev/cu.usbserial-2120", os.O_RDWR | os.O_NOCTTY | os.O_NONBLOCK)

Developer Workflow

Below are common commands and workflows for daily development.

Full Build and Flash

Standard Development Workflow 
# 1. Build the project
make -C apps/ipro7_demo

# 2. Flash to the development board
python3 tools/serial_monitor.py /dev/cu.usbserial-2120 --send reboot_dl --timeout 1
tools/ipro_iot_tool_lite/ipro_iot_tool_lite \
    --chipname ipro7 \
    --port /dev/cu.usbserial-2120 \
    --firmware apps/ipro7_demo/build/build_out/ipro7_demo_IPRO7.bin \
    --baudrate 921600 \
    --flash-pin 0x02

# 3. Monitor output
python3 tools/serial_monitor.py /dev/cu.usbserial-2120

Quick Retest (No Rebuild)

Reboot Board Only 
# Reboot and wait for results
python3 tools/serial_monitor.py /dev/cu.usbserial-2120 \
    --send reboot \
    --timeout 30 \
    --wait-for "ALL TESTS PASSED|SOME TESTS FAILED"

Clean Build

Clean and Rebuild 
# Clean build artifacts and rebuild
make -C apps/ipro7_demo clean
make -C apps/ipro7_demo

Automated Test Workflow

One-Command Testing (Build + Flash + Verify) 
# Full test cycle with report generation
tools/board_test.sh --project apps/ipro_unit_test --report test_results/

# Skip build, flash and test only
tools/board_test.sh --project apps/ipro_unit_test --skip-build

# Skip build and flash, test only
tools/board_test.sh --project apps/ipro_unit_test --skip-build --skip-flash
